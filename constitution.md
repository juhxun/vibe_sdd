## 문서 작성 원칙
이 프로젝트에서 생성되는 모든 문서(constitution, spec, plan, tasks, checklist, 분석 리포트 등)는 반드시 한국어로 작성한다.
모든 문서 작성은 명확한 한국어 문장으로 작성하고, 외국어 약어는 필요 시 한국어 설명을 함께 표기한다.

## 핵심 원칙

### 1. 스펙 우선
모든 기능은 구현 전에 스펙 문서로 정의한다. 스펙에 없는 기능은 구현하지 않는다.

지켜졌는지 판정하는 기준:
- 새 기능 구현 전에 spec.md에 기능 명세가 작성되었는가?
- 구현 내용이 spec.md의 범위 안에 있는가?
- 스펙에 없는 기능이 코드에 포함되지 않았는가?

### 2. 테스트 필수
핵심 기능(추가/완료 토글/삭제/남은 개수/필터)은 pytest 자동화 테스트로 검증한다. 구현 완료의 기준은 해당 테스트가 통과하는 것이다.

지켜졌는지 판정하는 기준:
- 핵심 기능마다 pytest 테스트가 존재하는가?
- pytest 실행 결과가 모두 통과하는가?
- 변경 전후에 테스트가 실행되고 결과가 기록되는가?

### 3. 설정 분리
데이터베이스 경로 등 환경설정 값은 .env 파일로 관리하고 저장소에 커밋하지 않는다. 필요한 환경변수는 .env.example 파일에 문서화한다.

지켜졌는지 판정하는 기준:
- .env 파일이 저장소에 커밋되지 않았는가?
- .env.example에 필수 환경변수가 문서화되었는가?
- 코드에 민감한 설정값이 하드코딩되지 않았는가?

### 4. 추적 가능한 커밋
모든 커밋 메시지 앞에 태스크 ID를 붙인다. 예: T03: 할 일 생성 API 구현

지켜졌는지 판정하는 기준:
- 커밋 메시지가 T##: 형식으로 작성되었는가?
- 커밋 메시지의 태스크 ID가 실제 작업 항목과 일치하는가?

### 5. 단순성
요구사항에 없는 기능이나 라이브러리를 임의로 추가하지 않는다.

지켜졌는지 판정하는 기준:
- 새 라이브러리나 패키지가 요구사항에 명시된 이유와 함께 도입되었는가?
- 구현된 기능이 명시된 요구사항과 일치하는가?
- 불필요한 복잡성이나 과도한 추상화가 없는가?

## 추가 제약
- 모든 문서와 의사소통은 한국어로 작성한다.
- 환경설정 값은 .env와 .env.example로 관리한다.
- 새로운 의존성은 명시적 사유와 검토 없이 추가하지 않는다.

## 개발 워크플로우
- 기능 구현 전 스펙 검토를 완료한다.
- 핵심 기능 변경 시 테스트를 먼저 작성하거나 갱신하고, 테스트가 통과해야만 구현을 완료로 간주한다.
- 커밋 전 변경 사항과 테스트 결과를 확인한다.
- 문서 변경 시 한국어 문서 기준을 유지한다.

## Governance
이 헌법은 프로젝트의 품질, 추적 가능성, 그리고 문서 일관성을 보장하기 위한 최상위 규칙이다.
헌법의 변경은 문서 수정, 영향 범위 검토, 그리고 관련 문서 동기화를 포함한 승인 절차를 거쳐야 한다.
모든 변경 사항은 관련된 spec, plan, tasks, checklist, 분석 리포트와 함께 일관되게 업데이트한다.
