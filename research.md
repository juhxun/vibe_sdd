# 연구 결과: 오늘의 할 일 웹 앱

## 결정 사항

### 1. 서버 구조
- FastAPI를 사용해 REST API와 HTML 진입점을 함께 제공한다.
- 라우팅은 app/main.py에 두고, 데이터 접근 로직은 app/crud.py로 분리한다.

### 2. 데이터 저장 방식
- SQLite를 사용한다.
- 데이터베이스 경로는 .env 파일의 DATABASE_URL 환경변수로 관리한다.
- 로컬 브라우저/기기 단위의 데이터 유지 방식을 우선 채택한다.

### 3. 프론트엔드 렌더링 방식
- 메인 페이지는 Jinja2 템플릿으로 제공한다.
- 목록 갱신과 사용자 상호작용은 바닐라 JavaScript + fetch로 처리한다.
- 별도 빌드 도구 없이 단일 HTML 템플릿으로 구현한다.

### 4. 검증 방식
- API 동작은 pytest와 FastAPI TestClient로 검증한다.
- 핵심 기능(생성/완료 토글/삭제/남은 개수/필터)은 자동화 테스트로 커버한다.

## 근거

- 요구사항은 단순한 할 일 관리 앱이므로 복잡한 분산 아키텍처보다 단일 서버 구조가 적절하다.
- SQLite는 개발·운영 모두에서 설정이 단순하고, 현재 범위에 충분히 적합하다.
- Jinja2 + 바닐라 JavaScript 조합은 빌드 도구 없이 빠르게 구현할 수 있다.

## 대안 검토

### 대안 1: React/Vue 같은 SPA 프레임워크
- 장점: 복잡한 UI 상태 관리에 유리하다.
- 단점: 현재 요구사항 대비 과도하게 복잡하고 빌드 도구가 필요하다.
- 결정: 채택하지 않음.

### 대안 2: Flask + 템플릿 렌더링
- 장점: 구현 경험이 익숙하다.
- 단점: FastAPI가 더 적합한 타입 안정성과 자동 문서화 지원을 제공한다.
- 결정: 채택하지 않음.

## 구현 주의사항

- 빈 제목은 API 레벨에서 422로 거절한다.
- 완료 토글과 삭제는 ID 기반으로 처리한다.
- 목록 필터는 query parameter status로 표현한다.
- 프론트엔드는 페이지 전체 새로고침 없이 fetch 기반으로 갱신한다.
