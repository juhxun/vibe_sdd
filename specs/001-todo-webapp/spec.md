# 기능 명세: 오늘의 할 일 웹 앱

**기능 브랜치**: `001-todo-webapp`

**Created**: 2026-07-15

**Status**: Draft

**Input**: 사용자 설명: "오늘의 할 일" 웹 앱을 만들려고 해. 사용자 관점의 요구사항은 다음과 같아.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 새 할 일 추가 (우선순위: P1)
사용자는 입력창에 새 할 일을 입력하고 추가 버튼을 눌러 목록에 포함시킬 수 있다. 빈 값이나 공백만 있는 입력은 추가되지 않으며, 사용자에게 안내 메시지를 확인할 수 있다.

**Why this priority**: 할 일 관리 앱의 핵심 가치는 새 항목을 빠르고 안전하게 추가하는 데 있으므로 가장 먼저 제공되어야 한다.

**Independent Test**: 사용자가 새 할 일을 입력하고 추가하면 목록에 즉시 반영되며, 잘못된 입력은 거절되고 안내 메시지가 표시되는지로 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 할 일 목록이 비어 있는 상태일 때, **When** 사용자가 유효한 내용을 입력하고 추가하면, **Then** 새 할 일이 목록에 추가되고 남은 할 일 수가 갱신된다.
2. **Given** 입력창에 공백만 입력된 상태일 때, **When** 사용자가 추가를 시도하면, **Then** 할 일이 추가되지 않고 안내 메시지가 표시된다.

---

### 사용자 스토리 2 - 완료 상태 관리 및 시각적 구분 (우선순위: P1)
사용자는 할 일을 완료 처리할 수 있고, 완료된 항목은 취소선과 흐린 색으로 표시된다. 완료된 항목은 다시 미완료 상태로 되돌릴 수 있다.

**Why this priority**: 사용자가 진행 상황을 쉽게 파악하고 상태를 전환할 수 있어야 앱의 핵심 사용 경험이 완성된다.

**Independent Test**: 사용자가 항목을 클릭하거나 토글하면 상태가 변경되고 시각적 표시가 바뀌는지로 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 미완료 상태의 할 일이 있을 때, **When** 사용자가 해당 항목을 완료 처리하면, **Then** 항목은 완료 상태로 바뀌고 취소선과 흐린 색으로 표시된다.
2. **Given** 완료 상태의 할 일이 있을 때, **When** 사용자가 해당 항목을 다시 미완료로 되돌리면, **Then** 항목은 원래 상태로 복구되고 남은 할 일 수가 다시 갱신된다.

---

### 사용자 스토리 3 - 삭제, 필터, 상태 표시 (우선순위: P2)
사용자는 할 일을 삭제할 수 있고, 삭제 전 확인 절차를 거친다. 화면 상단에는 남은 할 일 수가 표시되며, 전체/진행중/완료 필터로 목록을 볼 수 있다.

**Why this priority**: 삭제와 필터링은 효율적인 할 일 관리 경험을 위해 필요하지만, 기본 생성/상태 전환 기능보다 상대적으로 덜 긴급하다.

**Independent Test**: 사용자가 삭제 요청, 필터 선택, 그리고 남은 개수 확인을 수행했을 때 각 기능이 올바르게 동작하는지로 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 삭제하려는 할 일이 있을 때, **When** 사용자가 삭제를 시도하면, **Then** 확인 절차가 먼저 표시되고 확인 후 항목이 제거된다.
2. **Given** 여러 할 일이 있을 때, **When** 사용자가 전체/진행중/완료 필터 중 하나를 선택하면, **Then** 해당 조건에 맞는 항목만 목록에 표시된다.
3. **Given** 할 일 목록의 상태가 변경될 때, **When** 추가/완료/삭제 동작이 발생하면, **Then** 화면 상단의 남은 할 일 수가 즉시 갱신된다.

---

### Edge Cases

- 입력값이 비어 있거나 공백만 있을 때는 추가되지 않고 안내 메시지를 보여준다.
- 삭제 확인을 취소하면 할 일은 유지된다.
- 할 일이 하나도 없을 때는 빈 상태 안내 문구를 보여준다.
- 새로고침이나 서버 재시작 후에도 기존 할 일 목록이 유지되어야 한다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 텍스트 입력을 통해 새 할 일을 추가할 수 있어야 한다.
- **FR-002**: 시스템은 빈 값 또는 공백만 있는 입력을 거절하고, 사용자에게 안내 메시지를 표시해야 한다.
- **FR-003**: 시스템은 사용자가 할 일을 완료 또는 미완료 상태로 전환할 수 있어야 한다.
- **FR-004**: 시스템은 완료된 할 일을 취소선과 흐린 색으로 시각적으로 구분해야 한다.
- **FR-005**: 시스템은 사용자가 할 일을 삭제할 수 있고, 삭제 전 확인 절차를 제공해야 한다.
- **FR-006**: 시스템은 화면 상단에 남은 할 일 수를 항상 표시하고, 상태 변경 시 즉시 갱신해야 한다.
- **FR-007**: 시스템은 전체 / 진행중 / 완료 필터를 제공하고, 선택된 필터에 따라 목록을 보여줘야 한다.
- **FR-008**: 시스템은 브라우저 새로고침이나 서버 재시작 후에도 할 일 목록이 유지되도록 데이터를 영속화해야 한다.
- **FR-009**: 시스템은 360px 폭의 모바일 화면에서도 주요 기능을 사용할 수 있도록 반응형 레이아웃을 제공해야 한다.
- **FR-010**: 시스템은 할 일이 하나도 없을 때 친절한 빈 상태 안내 문구를 보여줘야 한다.
- **FR-011**: 시스템은 할 일 데이터의 저장 범위를 현재 브라우저/기기 내에서 유지하는 방식으로 정의한다.

### 핵심 엔티티

- **할 일 항목**: 사용자가 관리하는 개별 작업으로, 제목, 완료 여부, 생성 시점 정보를 포함한다.
- **필터 상태**: 사용자가 현재 보고 있는 목록 범위를 나타내는 값으로, 전체/진행중/완료 중 하나를 가진다.

## 결정 사항

- 할 일 데이터는 현재 브라우저/기기에서만 유지되는 로컬 저장 방식으로 정의한다.
- 별도 계정 연동이나 다중 기기 동기화는 이번 범위에 포함하지 않는다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 사용자가 새 할 일을 추가하고, 완료 상태를 변경하며, 삭제하는 주요 흐름을 3분 이내에 완료할 수 있다.
- **SC-002**: 브라우저를 새로고침하거나 서버를 재시작한 뒤에도 사용자가 이전 할 일 목록을 확인할 수 있다.
- **SC-003**: 360px 폭의 화면에서도 주요 기능에 접근하고 사용자가 할 일을 관리할 수 있다.
- **SC-004**: 할 일이 없을 때 사용자에게 명확한 빈 상태 안내가 표시되어, 다음 행동을 이해할 수 있다.

## 가정

- 사용자는 하나의 장치나 브라우저 세션에서 할 일 목록을 관리하는 기본 사용자다.
- 데이터 영속성은 사용자가 현재 브라우저와 서버 환경에서 기대하는 기본 수준으로 제공된다.
- 기술 스택과 저장소 방식은 계획 단계에서 결정한다.
- 현재 기능 범위는 할 일 생성, 상태 관리, 삭제, 필터링, 저장, 반응형 UI, 빈 상태 안내를 포함한다.
